# Forge Contract Proofs

Machine-checked proofs for the [Forge Perfect Loop](../docs/designs/forge-loop-contract-test-harness.md) delivery state machine, with extraction to a runnable OCaml checker.

## What is proven

The Coq development in `theories/` formalizes the canonical delivery states and transitions defined in `tools/contract-tests/fixtures/loop-contract.yaml`, then proves five Perfect Loop invariants **with zero `Admitted` proofs**:

1. **No dead-end non-terminal states** — every working state can advance.
2. **Terminal states are sinks** — `done` and `halted-*` states have no outgoing edges.
3. **Entry points terminate** — every fixture entry point (`in-analysis`, `ready-for-dev`, `ready-for-qa`, `ready-for-acceptance`) can reach `done`.
4. **Progress or halt or handback** — every transition either lands in a terminal state, makes forward progress (reduces rank), or is a bounded handback to an earlier pipeline stage.
5. **Done reachable from every non-terminal state** — the delivery pipeline cannot trap work (halted states are terminal sinks by design).

The composite theorem `forge_perfect_loop_well_formed` in `theories/Properties.v` bundles all five. A sixth *budget* note — `max_iterations` (12) exceeds the longest simple forward path, so the extracted checker's level-based BFS cannot false-negative within that budget — is documented in `Properties.v` but not stated as a theorem (it is a conservative constant, not a property requiring proof).

## What is NOT proven

- The LLM will follow `SKILL.md`.
- Natural-language handoff descriptions are semantically correct.
- Skill outputs (code, tests, acceptance criteria) are functionally correct.

Those require the runtime behavioral-evaluation layer (isolated subagent A/B tests). A theorem prover proves properties of the *specification*; it cannot prove properties of the model that interprets it.

## Project structure

```text
tools/contract-proofs/
├── dune-project          # opam package definition
├── Makefile              # build / check / test / extract targets
├── .gitignore            # excludes all generated artifacts below
├── theories/
│   ├── Generated.v       # GENERATED: State, Transition, successors, rank,
│   │                     #   all_states, entry_points, A1 reachability witnesses
│   ├── Transitions.v     # successors↔Transition agreement; terminals-are-sinks
│   ├── Paths.v           # max_iterations budget, progress-or-halt lemma
│   ├── Properties.v      # The Perfect Loop theorems + composite
│   ├── Extraction.v      # OCaml extraction (run via `make extract`)
│   └── dune              # Coq theory declaration
├── extraction/
│   ├── checker.ml        # GENERATED: BFS reachability checker iterating
│   │                     #   extracted all_states/entry_points (no hard-coded list)
│   ├── forge_state_machine.ml   # GENERATED by `make extract` (gitignored)
│   ├── forge_state_machine.mli  # GENERATED by `make extract` (gitignored)
│   └── dune                     # OCaml executable
└── test/
    ├── test_checker.ml          # Sanity tests for extracted checker
    ├── forge_state_machine.ml   # COPY of generated module for test build (gitignored)
    ├── forge_state_machine.mli  # COPY (gitignored)
    └── dune
```

All files marked **GENERATED** above are produced by `make` and are gitignored —
the repo carries only the generator (`tools/contract-tests/src/bin/gen_coq.rs`)
and the hand-written structural lemmas. A fresh clone builds from zero via `make`.

## Requirements

- OCaml >= 4.14
- Rocq/Coq >= 8.20 (tested with Rocq 9.1.1)
- Dune >= 3.17
- The Coq `Stdlib` library on the loadpath (`/opt/homebrew/lib/ocaml/coq/user-contrib/Stdlib` on Homebrew)

On macOS with Homebrew:

```bash
brew install opam
opam init
eval $(opam env)
opam install coq dune
```

## Build and check

```bash
cd tools/contract-proofs
make
```

`make` (i.e. `make all`) runs the pipeline in dependency order:

1. `make gen` — `gen_coq.rs` parses the fixture + skill handoff graph via `Repo::from_root` and emits `theories/Generated.v` (the proof model) and `extraction/checker.ml` (the runtime checker). Rust-only, no Python.
2. `make extract` — builds the Coq theories with dune, then runs `coqc Extraction.v` to emit `forge_state_machine.ml{,i}` into `extraction/` and `test/`.
3. `make build` — `dune build` compiles the proofs (proving all theorems) and links the extracted OCaml checker.
4. `make check` — `dune exec -- forge-state-machine` runs the checker.
5. `make test` — `dune runtest` runs the OCaml sanity tests.

Expected output from the checker:

```
OK: every non-terminal state has at least one successor.
OK: terminal states have no outgoing edges.
OK: every entry point can reach done.
All Forge Perfect Loop checks passed.
```

## Regenerating the generated artifacts

All generated files (`Generated.v`, `checker.ml`, `forge_state_machine.ml{,i}` and its `test/` copy) are gitignored. They are regenerated by `make` in the order above. To regenerate individually:

```bash
make gen       # regenerate Generated.v + checker.ml from the fixture
make extract   # regenerate forge_state_machine.ml{,i} from the proofs
make check-clean   # CI: exit non-zero if Generated.v or checker.ml are stale
```

`make check-clean` is the drift gate: it re-runs `gen_coq --check`, which exits non-zero (and prints `STALE`) if either emitted file would differ from what's on disk. Drift is caught a second way too — if the fixture changes such that a generated witness path no longer exists, `dune build` fails at the witness lemma (`The reference S_done was not found` or equivalent).

## Relationship to the Rust contract-test harness

```text
Layer 3: Behavioral evals (subagent A/B)        → empirical LLM correctness
Layer 2: Rust contract-test harness             → deterministic repo validation
Layer 1: Coq contract proofs (this directory)   → machine-checked specification
```

Layer 1 catches specification/harness bugs that Layer 2 would otherwise have to assert by test. When the canonical fixture changes, the Coq model must be updated and re-proven, creating a formal change-control gate.
